The sge universe

Introduction

This site exists to give you an overview of all the libraries and projects related to the graphics engine sge and also to give you some pointers on where to look if you want to get involved.

History

After meditating over Stroustrup's great tome "The C++ programming language" for over 9000 years, Freundlich started a one-man C++ project to develop some sort of "space game". Initially using Windows and DirectX, he quickly hacked together something useful, but in a one-man project, it's hard to keep motivating yourself.

So about a year after its inception, Phillemann joined sge and the project developed more quickly. Some time later, they both agreed to work on a game called sanguis together. This time, however, it wasn't supposed to be a space game. It was supposed to be a down shooter in the style of Crimsonland or violetland. This is when nille joined the team as a graphics and sound artist.

During the development of sanguis, it became apparent that sge was too monolithic. A lot of the functions and concepts were too general to be in a graphics engine. For example, sge had a logging facility, a math library and so forth. So a decision was made to move the general stuff to a new library, fcppt. Later on, a few other libraries were spawned, like awl and mizuiro.

Philosophy

There are a lot of graphics engines out there. Most prominently, there are irrlicht and ogre. But those aren't the only ones. So, should I choose sge over those two? The answer is: it depends.

The problem with virtually all of the open source graphics engines is that they don't use proper, modern C++. Yeah, they use classes and shit, but is the code exception-safe and memory-leak-free? Are the external libraries properly sealed off from the user via a clean, well-though interface? Or does it use singletons to make your life hard and mask dependencies between the subsystems, put everything into one big namespace and has needlessly huge base classes?

So sge tries to be distinct by being as clean, memory-leak-free, exception-safe, singleton-less and portable as possible. That, however, comes with a price:

  • sge ist very actively developed, which means that you have to be prepared for sudden changes in the API. We as developers are, of course, eager to explain why the changes were made and how you can adapt your code.
  • sge can — in its current state — more be thought of as a low-level library. For example, it has a very clean, abstract renderer interface and a good abstraction for vertex buffers, but it lacks the concept of a scene graph and all the advantages this brings. In this case, it's because implementing a scene graph "The Right Way" and being as abstract as possible is extremely difficult.
  • sge tries to implement everything as clean as possible, but it also does most things on demand. So if you're missing something in sge, we might not have needed it yet (in sanguis, for example).
  • There is no documentation available. This is also due to the volatile nature of the project. There are only two people working on it, so writing documentation for an interface that might change in the near future is pretty much useless. But: We have an IRC channel producing page-long logs each day and we love to talk about sge and its sister projects, so we're happy to answer all your questions. Also, we have at least one stand-alone example for every subsystem in sge, with code that's well-written and partially documented (we're working on the documentation, though).
So in summation, if you want to be as productive as possible with your project, you will most likely be happier with the larger engine projects. But then, you have to use their crappy interfaces.

What does sge offer?

  • A complete renderer interface with backends for DirectX (in development) and OpenGL (considered mature). There's a really nice (modern C++y) abstraction layer over vertex buffers. There is, however, no shader abstraction if you want to be independent of HLSL or GLSL.
  • A very abstract, yet optimized sprite system supporting point sprites.
  • The ability load truetype fonts (via freetype plugin) and bitmap fonts with an interface which is agnostic as to what specific type of font you're using.
  • An audio subsystem supporting streaming sounds (for background music) and nonstreaming sounds. With loaders for wave and Ogg/Vorbis and with an OpenAL player backend.
  • Abstractions for timers
